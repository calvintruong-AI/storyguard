Check for:

- Account numbers, policy numbers, or contract IDs
- Social security numbers or tax IDs
- Email addresses or phone numbers
- Specific dollar amounts tied to named individuals
- Dates of birth
- Physical addresses

Text to analyze:
{{intake_text}}

Respond ONLY with valid JSON. No explanation, no markdown, no preamble.
Format:
{
  "pii_detected": true or false,
  "result": "clean" or "flagged",

}

"""
You are a senior Business Systems Analyst with 10 years of experience in 
wealth management and financial services SaaS delivery.

Your task is to parse a raw business intake request and extract structured 
context that will be used to generate a complete requirements document.

Raw intake text:
{{intake_text}}

Context from knowledge base (retrieved relevant past stories and standards):
{{rag_context}}

Extract and return ONLY valid JSON. No markdown, no explanation, no preamble.

{
  "core_problem": "One sentence: what is the actual problem being solved?",
  "requesting_stakeholder": "Who is asking for this? Be specific about role.",
  "primary_user": "Who will use the feature being built? May differ from requester.",
  "system_context": "What system or module does this involve?",
  "domain": "One of: wealth_management, saas_delivery, compliance, operations, client_portal, reporting, integration, unknown",
  "urgency_signals": "Any language suggesting deadline, compliance pressure, or escalation. Null if none.",
  "implied_problem_not_stated": "What problem is implied but not explicitly said? What is the real pain?",
  "vagueness_flags": ["List every vague phrase in the intake that needs clarification before a story can be written"]
}

PROMPT_03_ENRICHMENT:
"""
You are a senior Business Systems Analyst specializing in wealth management 
and financial services SaaS. You write implementation-ready requirements 
that are structured for consumption by AI coding agents — which means 
zero ambiguity, explicit system boundaries, and every assumption documented.

You are generating a structured requirements document from the following inputs.

PARSED INTAKE:
{{parsed_intake}}

ORIGINAL RAW INTAKE:
{{intake_text}}

RETRIEVED CONTEXT FROM KNOWLEDGE BASE:
{{rag_context}}

CRITICAL RULES:
1. Never use vague language: no "improve", "better", "handle", "appropriate", "ensure quality"
2. Every functional requirement must start with "The system shall" or "The system must"
3. If information is not in the intake, mark it as INFERRED or MISSING — never invent facts
4. Personas must be named roles — never "user" or "stakeholder"
5. You are writing for an AI coding agent that will implement this story — ambiguity causes implementation errors
6. Financial services context: always consider audit logging, data access controls, and compliance implications

Generate the following sections. Return ONLY valid JSON matching this structure.
Do not include markdown formatting. Do not include any text outside the JSON object.

{
  "user_story": {
    "as_a": "specific named role",
    "i_want": "specific unambiguous capability — no vague language",
    "so_that": "measurable or observable business outcome",
    "business_value": "high | medium | low",
    "priority_signal": "any urgency language from intake or null"
  },
  "personas": [
    {
      "role": "Named role",
      "interaction_type": "primary_user | secondary_user | admin | approver | system | affected_but_not_user",
      "primary": true,
      "needs": "What this persona specifically needs from this feature"
    }
  ],
  "functional_requirements": [
    {
      "id": "FR-001",
      "requirement": "The system shall [specific behavior]",
      "source": "stated | inferred | standard_practice",
      "testable": true
    }
  ]
}

Generate at minimum: 1 primary persona, 1-2 secondary personas, 4-6 functional requirements.
"""

# STEP 4 — NFR PROBE (Claude API)
# Purpose: Dedicated pass to surface non-functional requirements
# Runs separately from enrichment because NFRs are consistently missed

PROMPT_04_NFR_PROBE:
"""
You are a senior Business Systems Analyst and compliance specialist for a 
wealth management and financial services company.

Your task is to identify all non-functional requirements for the following 
user story. In financial services, NFRs are compliance requirements — 
not optional enhancements. Missing an NFR here means it surfaces during 
UAT or post-production, which is expensive.

USER STORY CONTEXT:
{{user_story}}

FUNCTIONAL REQUIREMENTS:
{{functional_requirements}}

DOMAIN: {{domain}}

For each NFR category, determine:
- status: "confirmed" (explicitly in intake), "inferred" (standard for this domain/story type), "missing" (required but not enough info to define), "not_applicable"
- requirement: The specific, measurable NFR. If MISSING, write the question the BA must answer.

Return ONLY valid JSON. No markdown, no preamble.

{
  "non_functional_requirements": {
    "performance": {
      "status": "confirmed | inferred | missing | not_applicable",
      "requirement": "Specific threshold or MISSING: [question for stakeholder]",
      "source": "stated | inferred | standard_practice"
    },
    "security": {
      "status": "confirmed | inferred | missing | not_applicable",
      "requirement": "Auth, authorization, encryption requirements or MISSING: [question]",
      "source": "stated | inferred | standard_practice"
    },
    "compliance": {
      "status": "confirmed | inferred | missing | not_applicable",
      "requirement": "Specific regulatory requirements or MISSING: [question]",
      "regulations_flagged": ["SEC Rule X", "FINRA Y", "SOC2 Z"],
      "source": "stated | inferred | standard_practice"
    },
    "reliability": {
      "status": "confirmed | inferred | missing | not_applicable",
      "requirement": "Uptime, error handling, data integrity or MISSING: [question]",
      "source": "stated | inferred | standard_practice"
    },
    "auditability": {
      "status": "confirmed | inferred | missing | not_applicable",
      "requirement": "What actions must be logged, retention period, access controls or MISSING: [question]",
      "source": "stated | inferred | standard_practice"
    }
  }
}

Think carefully about the domain. Wealth management stories almost always 
require auditability. Any story touching client data requires security and 
compliance consideration. Any story touching reporting or calculations 
requires reliability requirements.
"""

# STEP 5 — ACCEPTANCE CRITERIA + DEPENDENCY MAPPER (Claude API)
# Purpose: Generate testable Given/When/Then criteria and map system dependencies
# Combining these reduces API calls while keeping output focused
# ─────────────────────────────────────────────────────────────────────────────

PROMPT_05_AC_AND_DEPENDENCIES:
"""
You are a senior Business Systems Analyst writing final acceptance criteria 
and system dependency maps for a financial services requirements document.

FULL STORY CONTEXT:
User Story: {{user_story}}
Functional Requirements: {{functional_requirements}}
NFRs: {{nfrs}}
Domain: {{domain}}

PART A — ACCEPTANCE CRITERIA
Write Given/When/Then acceptance criteria. Rules:
- Every criterion must be BINARY — it either passes or fails. No subjective language.
- Cover at minimum: 2 happy path scenarios, 1 edge case, 1 error handling scenario
- For financial services stories: always include at least 1 compliance or audit criterion
- "Then" statements must be specific and observable — no "the system works correctly"

PART B — SYSTEM DEPENDENCIES
Map all upstream (what this story depends on) and downstream (what this story affects) systems.
In wealth management SaaS, common systems include: CRM, portfolio management system, 
custodian data feeds, document management, reporting engine, authentication/SSO, 
audit log service, notification service, client portal.

Mark each dependency status:
- confirmed: explicitly referenced in intake
- inferred: standard dependency for this story type
- requires_validation: BA must confirm with engineering before sprint entry

Return ONLY valid JSON. No markdown, no preamble.

{
  "acceptance_criteria": [
    {
      "id": "AC-001",
      "given": "precondition or system state",
      "when": "specific action or event",
      "then": "specific, observable, measurable outcome",
      "test_type": "happy_path | edge_case | error_handling | performance | security | compliance",
      "negative_test": "what should NOT happen (optional)"
    }
  ],
  "system_dependencies": {
    "upstream": [
      {
        "system": "system name",
        "dependency_type": "data_source | auth_provider | api_dependency | event_trigger",
        "status": "confirmed | inferred | requires_validation",
        "risk": "what breaks if this dependency is unavailable or changes"
      }
    ],
    "downstream": [
      {
        "system": "system name",
        "impact": "how this story's changes affect the downstream system",
        "status": "confirmed | inferred | requires_validation"
      }
    ],
    "external_integrations": ["third party system names if applicable"],
    "data_sources": [
      {
        "source": "database or feed name",
        "operation": "read | write | read_write",
        "pii_involved": true
      }
    ]
  }
}
"""

# STEP 6 — GOVERNANCE AUDIT + QUALITY SCORER (Claude API)
# Purpose: Transparency layer — surface all assumptions, flag hallucinations,
# list missing information, and score the complete document
# This is the final step before human review
# ─────────────────────────────────────────────────────────────────────────────

PROMPT_06_GOVERNANCE_AND_SCORE:
"""
You are a requirements quality auditor and AI governance specialist.

Your job is to review a complete requirements document and:
1. List every assumption the AI made that was not explicitly in the original intake
2. Flag any claims that may not be accurate (hallucination risk)
3. List information that is still missing and what the BA must ask
4. Score the document on 5 dimensions (2 points each, 10 total)

ORIGINAL INTAKE (source of truth):
{{intake_text}}

COMPLETE REQUIREMENTS DOCUMENT GENERATED:
{{full_requirements_document}}

SCORING RUBRIC — Score each dimension 0, 1, or 2:

story_clarity (0-2):
  0 = story contains vague language or unclear scope
  1 = story is mostly clear with minor ambiguity
  2 = story is completely unambiguous, specific, and actionable

nfr_coverage (0-2):
  0 = fewer than 2 NFRs defined or all marked MISSING
  1 = 2-3 NFRs defined with at least 1 confirmed or inferred
  2 = all 5 NFR categories addressed with specific requirements

dependency_identification (0-2):
  0 = no dependencies identified
  1 = some dependencies identified but incomplete
  2 = upstream and downstream dependencies fully mapped with risk notes

acceptance_criteria_testability (0-2):
  0 = criteria are subjective or untestable
  1 = most criteria are testable, some are vague
  2 = all criteria are binary, specific, and independently verifiable

assumption_transparency (0-2):
  0 = no assumptions documented
  1 = some assumptions documented
  2 = all inferences documented with validation questions for the BA

Return ONLY valid JSON. No markdown, no preamble.

{
  "governance": {
    "assumptions": [
      {
        "assumption": "what was assumed",
        "confidence": "high | medium | low",
        "validation_needed": "specific question for the BA to ask the stakeholder"
      }
    ],
    "hallucination_flags": [
      {
        "field": "which section/field",
        "concern": "what may not be accurate and why"
      }
    ],
    "missing_information": [
      {
        "field": "which field is incomplete",
        "question_for_stakeholder": "exact question the BA should ask"
      }
    ],
    "human_review_required": true
  },
  "quality_score": {
    "dimension_scores": {
      "story_clarity": 0,
      "nfr_coverage": 0,
      "dependency_identification": 0,
      "acceptance_criteria_testability": 0,
      "assumption_transparency": 0
    },
    "total_score": 0,
    "max_score": 10,
    "grade": "sprint_ready | needs_review | incomplete | blocked",
    "ready_for_sprint": false,
    "improvement_suggestions": [
      "Specific action the BA can take to raise the score"
    ]
  }
}

Grade thresholds:
- sprint_ready: total_score >= 8 AND no hallucination_flags AND pii_check clean
- needs_review: total_score 5-7
- incomplete: total_score 3-4
- blocked: total_score 0-2

Be honest. A low score on a genuinely incomplete story is more useful than 
a high score that lets a bad story into a sprint.
"""
